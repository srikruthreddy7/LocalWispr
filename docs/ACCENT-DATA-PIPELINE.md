# Accent Data Pipeline

This is the working pipeline for Indian-accent English ASR data. The goal is narrow:

- improve accent understanding
- keep Svarah evaluation-only
- avoid training on formatting-heavy rows just because they are easy to score
- run independent audits, gated probes, and manifest builds in parallel whenever Modal quota allows

The source registry and command runbook are generated by:

```bash
python3 tools/accent_data_pipeline.py --output-dir artifacts/accent-data-pipeline
```

Quick local wiring check:

```bash
python3 tools/accent_data_pipeline.py --check
```

## Source Tiers

Tier 1:

- Indic-TIMIT in Modal: available now, strongest accent-focused corpus we currently control.
- `ishands/commonvoice-indian_accent`: open Hugging Face pool, about `163.9h`, best immediate open source candidate.
- IITM / NLTM Indian English ASR Challenge data: request/licensed source, likely the most important gated/request candidate after Indic-TIMIT.

Tier 2:

- `ARTPARK-IISc/Vaani-transcription-part`: gated access confirmed in Modal, broad India coverage, target the English config directly.
- `ARTPARK-IISc/Vaani`: gated, larger fallback when the transcription-part repo is not enough.
- `WillHeld/india_accent_cv`: known open baseline pool, useful for dedupe and regression checks.
- `raianand/TIE_shorts`: public academic Technical Indian English subset. Treat as lecture/technical-domain data until audited.

Tier 3:

- `KoelLabs/L2Arctic`: high-quality non-native English, small Hindi L1 subset, non-commercial.
- `skit-ai/skit-s2i`: small Indian-English telephony side pool, non-commercial.
- NPTEL2020: huge open lecture-style source, useful only with strict audit/teacher filtering.

### Manifest Normalization Fix

`tools/modal_mix_training_manifests.py` now canonicalizes mixed rows to the common local-loader fields:

- `audio`
- `text`
- `source_audio_column`
- `source_text_column`

This matters because Indic-TIMIT rows use `audio_path`/`transcript`, while CV and Vaani rows use `audio`/`text`. Without canonicalization, a mixed manifest can silently train against the wrong fields.

### Safe Aggregate Report Summaries

`tools/modal_artifact_report_summary.py` was added for gated artifacts. It reads `report.json` inside the Modal artifacts volume and prints only aggregate-safe fields:

- dataset metadata
- candidate labels and adapter run IDs
- overall WER/CER
- deltas vs base
- pairwise improved/unchanged/worsened counts
- progress state

## April 29 Gradient Fix And Fixed-Adapter Results

Root cause found:

- The first April 29 encoder-only LoRA adapters looked trained but behaved exactly like base on a 64-row Common Voice heldout smoke test.
- Training logs showed `grad_norm: 0.0` plus the PyTorch warning that inputs had no gradients.

### Cohere LoRA and vLLM support

Current vLLM support status:

- vLLM lists `CohereAsrForConditionalGeneration` for `CohereLabs/cohere-transcribe-03-2026`.
- The LoRA support column for Cohere ASR is blank in the vLLM supported-model table.
- Voxtral and Granite Speech have LoRA support checkmarks; Cohere ASR does not.

Decision:

- dynamic Cohere LoRA adapters are not supported by vLLM today
- `vllm serve CohereLabs/cohere-transcribe-03-2026 --enable-lora --lora-modules accent=...` should not be treated as a viable path
- if we LoRA Cohere and insist on live adapters, we are back on HF Transformers/PEFT, which we measured as slower than Whisper turbo
- the practical speed-preserving path is:
  1. train Cohere LoRA with HF/PEFT
  2. merge the adapter into the base weights
  3. save a full derived checkpoint
  4. try serving the merged checkpoint in vLLM
  5. benchmark against base Cohere vLLM and Whisper turbo

Engineering effort estimate for true dynamic adapter support inside vLLM:

| Path | Estimated effort | Read |
| --- | ---: | --- |
| Merge LoRA into Cohere weights and serve merged checkpoint | `1-2` days to prove | best practical path |
| Add decoder-only dynamic LoRA to Cohere vLLM | `3-7` days | may not help accent much |
| Add encoder+decoder dynamic LoRA for Cohere ASR | `2-4` weeks | real vLLM feature work |
| Upstream-quality PR with tests/docs | `4-6` weeks | requires maintainer review |

Why dynamic support is not trivial:

- Cohere Transcribe is a `2B` Conformer encoder-decoder ASR model, not a plain decoder-only LLM.
- Accent improvement likely needs LoRA in the audio/acoustic encoder, not only the text decoder.
- vLLM LoRA support is model-specific and the model class must declare/wire supported modules.
- A partial implementation could run while silently producing wrong outputs or losing the speed advantage.

Voxtral Mini 4B Realtime read:

- single-L40S viable: checkpoint size `8.25 GiB`, observed model memory about `8.38 GiB`
- vLLM detected architecture `VoxtralRealtimeGeneration`
- server exposes `/v1/realtime`
- accuracy is not acceptable for Svarah English
- failure mode: the model often emits phonetic English in non-Latin script

- `mistralai/Voxtral-Small-24B-2507` is not the cheap path. Its model card says BF16/FP16 requires about `55 GB` GPU RAM, so it does not fit on one L40S.

- These failures are access/infrastructure failures, not benchmark results.

Final May 1 decision:

- For highest accuracy, `openai/whisper-large-v3` remains the measured best single model on Svarah.
- For speed, Cohere vLLM is the measured best practical single-GPU backend, but it loses on accuracy.
- Cohere dynamic LoRA adapters are not supported by vLLM today.
- The only Cohere fine-tuning path that preserves vLLM speed is likely LoRA training followed by adapter merge into a full checkpoint.
- Voxtral Mini 4B Realtime is cheap and runnable on L40S, but its script/language behavior makes it unsuitable for this Indian-accent English benchmark right now.
- Do not start more Whisper LoRA runs until the next experiment has a new signal: confidence/logprob routing, merged Cohere proof, or a better non-Svarah data source.
